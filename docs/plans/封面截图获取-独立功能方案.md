# 功能方案:封面 / 截图 独立获取

> 状态:**进行中**(阶段 1-2 已落地,2026-06-18)
> 创建日期:2026-06-17
> 起因:当前封面只有两个分离入口——刮削时下载、或 ffmpeg 从视频截帧;没有「独立、多来源、可换图」的封面/截图获取功能。受 JAV-Preview 启发(DMM URL 直拼零爬取),把图片获取抽成独立能力。

---

## 1. 现状

| 来源 | 入口 | 问题 |
|---|---|---|
| 刮削下载 | `rs_scrape_save` 内 | 与刮削绑定,不能单独「只换封面」 |
| 视频截帧 | `capture_video_frames` + `save_captured_cover` | 手动截取,与刮削是两套独立 UI |
| 远程图下载 | `download_remote_image` / `rs_proxy_image` | 底层能力,无「按番号自动取图」的上层功能 |

→ 已有底层能力齐全(下载/代理/截帧/保存/删除:`media/commands.rs`),**缺一个统一的「图片获取中心」**,尤其缺 DMM 直拼这种快源。

## 2. 目标

独立的「获取封面 / 截图」功能:
- 对任一视频(有番号即可),**一键从多来源聚合候选**,预览后选定设为封面/截图。
- 不必走完整刮削;支持**单个换图**与**批量补全**。
- 来源:DMM 直拼(零爬取) + 刮削源 + ffmpeg 截帧。

## 3. 来源设计(按速度/质量优先级)

| 优先级 | 来源 | 封面(poster) | 截图(thumbs) | 说明 |
|---|---|---|---|---|
| 1 | **DMM 直拼**(见 competitor-analysis/jav-preview.md) | `pics.dmm.co.jp/.../{cid}pl.jpg` | `pics.dmm.co.jp/.../{cid}jp-{n}.jpg` | 零爬取、官方高清、无地理封锁;**有番号即可** |
| 2 | 刮削源 | 各源 cover/poster | 各源 thumbs | 复用现有 16 源,覆盖广 |
| 3 | ffmpeg 截帧 | 单帧 | 多帧 | 本地兜底,无网络/无官方图时(现有 `capture_video_frames`) |

- DMM 仅覆盖有码主流(FANZA);无码/FC2/素人回退刮削源或截帧。
- DMM 清晰度/后缀因片而异,需 **HEAD 探测**哪个 URL 可用(`pl`/`ps`、`jp-1..N`)。

## 3.5 横竖封面规则(关键:poster 竖 / fanart 横)

> ✅ **本节的图集生成已于 2026-06-18 随「NFO 独立存储与标准化」落地**:`media/artwork.rs` 产出 poster(竖)/fanart(横)/thumb(横)+extrafanart,横版右裁出真实竖版;前端 `resolveCoverImage` 按 `coverType` 选图。本方案剩余范围 = DMM 直拼快源 + 多来源候选聚合 + 换图/批量补全 UI(图集落地复用 `artwork.rs`)。详见 `nfo-独立存储与标准化-计划.md` 的「图集标准化」一节。


**媒体库规范**(Kodi/Emby/Jellyfin):`poster`=竖版(2:3,海报墙主图)、`fanart`=横版(宽幅,详情页背景),**用途不同、都要存**;媒体库识别**优先靠同目录文件名约定**(`{stem}-poster.jpg` 竖 / `{stem}-fanart.jpg` 横 / `{stem}-thumb.jpg` 横缩略),NFO 的 `aspect` 为辅。

**JAV 特殊性**:官方封面(DMM `pl.jpg`)是**横版**大图;竖版 poster 的行业做法是从横版**右半裁剪**(`constants.ts` 已有此认知:portrait≈378×538=DMM 封面右侧裁切)。故**横版对 JAV 原生可靠,竖版需加工、可能失败**——这也是普通电影默认竖版、JAV 默认横版的根因(数据源性质不同)。

**获取策略(横竖都存,不二选一)**:
- 横版(→ `fanart`):JAV 几乎必得,直接拿,作**保底**。
- 竖版(→ `poster`)按优先级:① **刮削源直接提供的竖版** → ② 从横版**裁剪**(兜底,可能不完美)→ ③ 横版占位。
- 来源对应:DMM `pl` = 横版(作 fanart);DMM 无现成高清竖版,竖版需裁剪;**部分刮削源能直接给竖版 poster,优先用**。

**显示 fallback(任何布局都不留空白)**:
- 显示竖版(海报墙):竖版 poster → 横版裁剪 → 横版 `object-position` 截右侧。
- 显示横版(卡片):直接 `fanart`/`cover`。
- `coverType`(landscape/portrait)只决定**前端显示偏好**;两种图都该备好,缺一种回退另一种,再不行截帧。

**真实文件 vs 显示裁切**:外部媒体库要的是**真实的竖版 poster 文件**——前端 CSS 裁切不算数。要竖版海报墙在 Emby/Kodi 生效,需**实际生成裁剪后的 poster.jpg**。

**NFO / .strm**:NFO 用 `aspect` 标注(`<thumb aspect="poster">` 竖、`<fanart>` / `aspect="landscape"` 横,见 `nfo-独立存储与标准化-计划.md` 补 `<fanart>`);`.strm` **不含图片**,横竖图靠同目录文件名约定(独立目录存储方案需在 `.strm` 旁放好命名正确的横竖图)。

## 4. 交互设计

- **单视频换图**:视频卡/详情页「获取封面/截图」按钮 → 候选弹窗(按来源分组:DMM / 刮削 / 截帧)→ 选 1 张设封面 / 多张设截图 → 保存替换。
- **自动**:一键「自动取最优」按优先级取(DMM→刮削→截帧)。
- **批量补全**:媒体库多选「缺封面的视频」→ 批量自动获取(DMM 优先,失败回退截帧),进度展示(复用并发方案)。
- 截图:DMM `jp-{n}` 序列 / 刮削 thumbs / 截帧多帧,统一进截图候选。

## 5. 改动 / 复用

| 层 | 改动 |
|---|---|
| 新增 | `image_fetch` 模块:番号→DMM cid 转换 + URL 模板 + HEAD 探测可用性 |
| 新增 command | `get_image_candidates(video_id)` → 聚合各来源候选(DMM 拼好+探测、刮削、截帧选项) |
| 复用 | `download_remote_image` / `rs_proxy_image`(取图)、`capture_video_frames`(截帧)、`save_captured_cover` / `save_captured_thumbs`(保存)、`delete_cover` / `clear_thumbs`(替换前清理) |
| 前端 | 候选预览弹窗组件;视频卡/详情页入口;批量补全入口;`coverVersions` 刷新(已有) |

## 6. 分阶段

1. ✅ **DMM 直拼器**(`media/dmm.rs`):`designation_to_cid`(番号→cid,字母前缀+数字补零 5 位)+ digital/mono 海报 URL + 截图 jp-1..N + HEAD 探测(digital 优先回退 mono、截图连续探测遇缺即止)。单测覆盖 cid 转换/URL 模板;已对真实 DMM CDN 验证(`ssis00001pl.jpg` → 200)。
2. ✅ **候选聚合 command**(`get_image_candidates(video_id)`):查番号 → DMM 探测 → 返回 `ImageCandidate { source, kind: cover/screenshot, url }`。已注册。**后续接刮削源、ffmpeg 截帧候选。**
3. ✅ **单视频换图 UI**(`ImageFetchDialog.vue` + `apply_image_candidates` 命令):候选弹窗(DMM 官方图直接 `<img>` 预览,`no-referrer`)→ 选 1 封面 + N 截图 → 应用。封面走 `produce_artwork(dmm.cover_url)` 产 fanart+thumb+裁 poster 并写库;截图追加到 extrafanart。入口在视频详情封面右键菜单 +「暂无封面」按钮;应用后刷新封面与预览。
4. ✅ **批量补全**(`batch_fetch_covers` 命令 + 媒体库工具栏「批量获取封面」):抽出 `apply_images` 核心供单视频与批量共用;新增 `probe_dmm_cover`(仅探海报、轻量);批量对**当前筛选视图里缺封面的视频**有界并发(3)从 DMM 补全封面,进度经 `batch-fetch-cover-progress` 事件推送,完成汇总 + 刷新。DMM 无匹配则跳过(无码/素人留给截帧)。**后续可加 DMM→刮削→截帧 的多源优先级编排。**
5. ⬜ **接入探索/预览**:DMM 海报/截图同时服务探索模块的图片展示(见探索方案)。

## 7. 风险

- DMM 仅覆盖有码主流;番号→cid 有特例(前缀映射/补零/digital vs mono),需维护规则(参考 jav-preview.md)。
- DMM URL 后缀需探测,增加少量 HEAD 请求。
- 替换封面需正确清理旧文件 + 刷新 `coverVersions` 缓存键。
- **横版裁竖版 poster 可能裁歪/裁错**(无人脸对齐时):**仅作兜底,非主路径**;优先用源直取的竖版,裁剪失败回退横版显示,不留空白。
- **批量并发 + 独立目录 + 同番号多分片**:多个同番号视频(如 CD1/CD2)在独立目录模式下落到同一子目录、同 stem,批量并发(3)时可能同时写同一图片文件 → 撕裂。属罕见组合(跟随视频模式不受影响,因 stem 各异),重新应用即修复;如需可加按目标串行或原子写(临时文件 + rename)硬化。

## 代码审查修正(2026-06-18)

- **审查确认 DMM 探测正确**:DMM 缺失图返回 `302 → now_printing` 占位图,但 `wreq` 默认**不跟随重定向**(项目刻意如此,3xx=反爬/年龄门信号),故 `head().is_success()` 见到原始 302 → 正确判为「不存在」,不会误返回占位图。已实测 cover(200 直返)与缺失截图(302)行为。
- **修复:应用封面后回填封面尺寸**:`apply_images` / 截帧落地 / 扫描截帧 dispatcher 原先只写 poster/thumb/fanart 不写 `cover_width/height`,导致瀑布流比例用兜底值(独立目录模式扫描读不到尺寸,更是一直不回填)。抽出 `artwork::read_image_dimensions`,`update_video_cover_paths` 补 `cover_width/height` 参数,四处调用统一回填。
