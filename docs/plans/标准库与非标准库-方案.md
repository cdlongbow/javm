# 功能方案:标准库 / 非标准库 双模式

> 状态:**已落地**(2026-06-20):按「番号有无」派生分类,媒体库筛选「标准库/非标准库」+ 库健康「缺番号」一键跳非标准库;转正/手动编辑复用既有详情页能力。
> 创建日期:2026-06-17
> 优先级:🟡 中(JvedioNext #2 标 🔴)
> 目标:番号识别失败的资源(合集、散装、无番号、特殊命名)不再被丢弃,进入「非标准库」单独管理,可手动归正。

---

## 实现纪要(2026-06-20)

落地时核对代码实情,对原设计做了关键简化:

- **现状校正**:本应用扫描**从不丢弃**任何视频——无规范番号的文件照常入库(`local_id` 为空、`scan_status=1`);`scan_status=3`(识别失败)在全代码库**从未被赋值**(仅库健康统计读取)。故「不丢弃」目标本就满足,缺的只是**把无番号资源单列出来**的视图。
- **分类派生而非建列**:**非标准库 = `local_id` 为空(无番号)**,标准库 = 有番号。直接由 `local_id` 派生,**不新增 `library_type` 列**——避免再引入一处需与扫描流程同步、易出保留态 bug 的状态(独立目录封面曾因类似同步问题被扫描覆盖)。「用户手动归类」亦可表达:清空番号→非标准、填番号→标准,语义自洽。
- **改动**:`VideoFilter.libraryType`(`standard`/`nonStandard`)+ store 派生过滤 + 媒体库筛选面板「库类型」勾选(镜像有码/无码);库健康「缺番号」卡 → 「非标准库」按钮跳转过滤。**无后端/无 schema 改动**。
- **转正与编辑复用既有**:详情页已支持编辑番号、正则 `autoRecognizeLocalId`、AI 识别、按番号刮削——填/识别番号后刮削,`local_id` 落库即**自动转入标准库**(派生分类无需手动迁移)。

## 1. 现状

- 识别流程番号驱动:`designation_recognizer` 识别失败(`scan_status=3`)即「识别失败」,无后续去处,资源等于被搁置/丢弃。
- 现实中大量资源无规范番号:多集合集、个人/素人散装、欧美/国产、改名文件。

## 2. 设计:两类库

| | 标准库 | 非标准库 |
|---|---|---|
| 进入条件 | 番号识别成功 | 识别失败 / 无番号 / 用户手动归类 |
| 元数据 | 正常刮削 | 不强制番号;可手动填标题/演员/标签,或手动指定番号后「转正」 |
| 浏览 | 正常维度浏览 | 独立列表,支持搜索/标签/手动整理 |

**关键:识别失败 → 自动进非标准库,不丢弃。** 用户可:
- 手动指定正确番号 → 重新识别刮削 → **转入标准库**。
- 不指定番号,仅手动打标题/演员/标签,留在非标准库管理。

## 3. 复用 / 改动

| 层 | 改动 |
|---|---|
| `db/` | `videos` 加 `library_type`(standard / non_standard);识别失败默认 non_standard |
| 后端 | 手动指定番号 command(→ 重识别/刮削/转正);非标准库手动编辑元数据 |
| 前端 | 非标准库视图(列表 + 手动整理 UI);从诊断面板「识别失败」可跳入 |
| 复用 | 现有识别/刮削/手动编辑;与诊断面板联动 |

## 4. 分阶段

1. ✅ **不丢弃 + 独立列表**:无番号资源本就入库,新增「库类型」筛选单列出非标准库。
2. ✅ **手动指定番号 → 转正**:详情页填/识别番号 → 刮削 → `local_id` 落库自动转标准库。
3. ✅ **非标准库手动编辑**:复用详情页手动编辑元数据/标签 + 独立筛选浏览。

> 注:原计划的「`library_type` 列 / 识别失败默认 non_standard」未采用,改为按番号有无派生(见上「实现纪要」)。

## 5. 风险

- 非标准库可能积累大量杂项 → 提供批量操作/隐藏。
- 「转正」需校验番号正确性。
