# 对标分析:PornBoss

- **仓库**:https://github.com/Solr159/PornBoss
- **分析日期**:2026-06-17
- **分析维度**:刮削 / 番号识别 / 多源合并 / 架构
- **规模**:Go + React,~63 星,活跃维护(本日仍有提交)

## 一、定位

**迄今最接近本项目的同类品**:本地 JAV/视频管理桌面工具。后端 Go(Gin + GORM + SQLite),前端 React,内置 mpv 播放器。功能链路:目录扫描 → 番号识别 → 多源刮削 → 封面截图 → 入库 → 多维检索 + 播放。整体工程化程度高,数据源模块**每个都带单元测试**。

## 二、能力对比

| 维度 | 我们 (JAVManager) | PornBoss |
|---|---|---|
| 番号识别 | 5 类正则 + **AI** + 校验 + 规范化 | 双轨正则(**有码/无码分开**),含后缀字母、多分隔符,无 AI |
| 数据源 | 自研 16 源 | 6 源(javbus/avmoo/avsox/javdb/javdatabase/theporndb) |
| 多源结果处理 | `detail_score` **整条择优** | **字段级分工补全**(见提升点 #1) |
| 有码/无码 | 不区分 | 区分,`uncensored` 字段 + 无码专用源(avsox/avmoo) |
| 作品↔文件 | 一文件一记录(近期加去重) | **指纹 + 作品-文件解耦**(一个番号对应多个视频位置) |
| 女优维度 | 演员仅名字 | 女优视角 + 资料(身高/三围/生日)+ 别名合并 |
| 播放器 | Tauri WebView | 内置 mpv(高码率原生播放) |
| NFO | 生成 30+ 字段 | 支持情况未明确(代码有 nfo 引用,未深究生成/导入) |

## 三、可借鉴提升点

| # | 提升点 | PornBoss 的做法 | 我们现状 | 建议 | 优先级 |
|---|---|---|---|---|---|
| 1 | **字段级跨源分工补全** | `scanMissing*` 系列增量扫描:javdatabase 补英文/厂商,avmoo 补系列,javbus 补无码状态,avsox 补无码作品资料,另有中文补全——**每源补自己擅长的字段** | 多源 `detail_score` 整条择优,只留单一源 | 主源选定后按字段特长用其余源补全(**与 mediamatrix 的 `_merge` 互相印证,双竞品都这么做**) | 🔴 高 |
| 2 | **有码/无码分轨识别 + 无码专用源** | 两套番号正则(无码格式不同,如纯数字 `010120-001`)+ `uncensored` 字段 + avsox/avmoo 无码源 | 不区分,统一正则 | 番号识别区分有码/无码,无码走对应源 | 🟡 中 |
| 3 | **番号识别鲁棒性细节** | `([a-z]{2,6})[-_ ]?(\d{2,5})([a-z]{0,2})`:**捕获后缀字母**(ABC-123**A** 分片/版本)、**空格/下划线分隔**、返回多候选去重 | 用 `[_-]`,无后缀字母、无空格分隔 | 补后缀字母、空格分隔;保留我们的 AI(它没有) | 🟡 中 |
| 4 | **指纹 + 作品-文件解耦** | `fingerprint` + `video_location_jav_linker`:一个番号(作品)关联多个视频位置(多盘/NAS/移动盘的同作品多文件) | 近期才加去重,仍偏一文件一记录 | 引入「作品(番号) 1—N 文件位置」模型,天然支持去重与多副本管理 | 🟡 中 |
| 5 | **增量目录同步 + 失效标记** | `directory_scanner` 增量 upsert,文件大小+mtime 跳过未变,移除项标记 stale 隐藏 | 全量扫描 | 增量同步 + 失效隐藏(性能 + 实时性) | 🟢 中低 |
| 6 | **女优维度 + 资料合并** | 女优视角浏览 + 身高/三围/生日 + 别名 merge | 演员仅名字 | 女优维度 + 资料(呼应竞品分析的演员头像点) | 🟡 中 |
| 7 | **数据源解析单元测试** | 每个源 `*_test.go` | 部分(proptest/tempfile) | 给各数据源解析补测试,站点改版可回归 | 🟢 低 |
| 8 | 内置 mpv 播放器 | mpv 原生高码率播放 | Tauri WebView | 架构差异大,产品方向参考,不建议直接借鉴 | ⚪ 参考 |

## 四、关键结论

1. **最高价值、且被两个独立竞品印证的:字段级跨源分工补全(#1)。** PornBoss 用 `scanMissing*` 把它做成了规模化的「缺啥字段→用对应源补」,比 mediamatrix 的一次性 `_merge` 更工程化。我们应在 `resource_scrape` 结果合并层引入。
2. **番号识别互有长短**:我们有 AI + 校验(它没有),它有有码/无码分轨 + 后缀字母 + 多分隔符鲁棒性(我们没有)。可吸收它的细节(#2/#3),保留我们的 AI。
3. **架构启发**:指纹 + 作品-文件解耦(#4)是比"去重"更根本的模型,值得中期考虑。
4. NFO 我们仍领先(明确生成 30+ 字段);它的 NFO 支持不明确。
