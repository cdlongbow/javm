# 对标分析:mediamatrix-jav-expert

- **仓库**:https://github.com/emmylord/mediamatrix-jav-expert
- **分析日期**:2026-06-17
- **分析维度**:解析 / 番号识别 / 多源合并
- **规模**:Python,~2 星,MediaMatrix 的插件(`BasePlugin`)

## 一、定位

小型 Python 刮削插件,双源:**JavDB(社区数据)+ DMM(日本官方,需日本 IP/代理)**。核心就是「文件名 → 番号 → 双源刮削 → 合并 → NFO」。整体规模远小于本项目,多数能力不如我们,但其**字段级合并**设计值得借鉴。

## 二、解析能力对比

| 维度 | 我们 (JAVManager) | mediamatrix-jav-expert |
|---|---|---|
| 番号正则 | **5 类**(FC2/标准/混合前缀 T28/无连字符/纯数字),无连字符自动规范化 `ABC123→ABC-123` | 3 类(FC2/标准/无连字符) |
| AI 识别 | ✅ 有 | ❌ 无 |
| 番号校验 | ✅ `is_valid_designation`(前缀 2-6 / 数字 3-8) | ❌ 无,仅靠正则边界 |
| 数据源 | 自研 16 源 | 2 源(JavDB + DMM) |
| 多源结果处理 | 评分排序**整条择优**(`detail_score`),取单一最优源 | **字段级合并**(见下) |
| NFO 字段 | 30+ | 基础 |

→ 番号识别我们更强,无需借鉴。**唯一值得拿的是「多源结果处理」方式。**

## 三、可借鉴提升点(聚焦解析)

| # | 提升点 | 它的做法 | 我们现状 | 建议 | 优先级 |
|---|---|---|---|---|---|
| 1 | **字段级跨源融合** | `_merge`:以基础源为底,按字段特长用其它源覆盖——**评分/分类恒用社区源**(官方源没有),**官方标题/封面/演员名用官方源**,缺失字段互补 | 多源 `detail_score` **整条择优**,最终只来自单一源,其它源字段被丢弃 | 选定主源后,用其它源**补缺失字段**;关键字段按可信度合并,产出比任何单源都全的记录 | 🔴 高 |
| 2 | **按源特长的字段优先级** | 配置 `priority` + per-field 规则(评分锁定 JavDB,标题/演员锁定 DMM) | 只有整源评分,无 per-field 信任规则 | 配置化「字段→优先源」映射(如评分→社区源、官方名→官方源、封面→高清源) | 🟡 中 |
| 3 | DMM content_id 映射策略 | `cid_strategy: pattern_then_search`(番号→DMM cid 三种策略) | — | 将来接 DMM / r18.dev 官方源时参考 | 🟢 低(取决于是否接官方源) |
| 4 | 番号边界防误匹配 | 否定环视 `(?<![A-Za-z0-9])…(?![A-Za-z0-9])` 防从长串误提取 | Rust regex **不支持 lookbehind**,改用校验+位置兜底 | 引擎限制无法直接照搬,现有校验已可控,价值低 | ⚪ 可忽略 |

## 四、关键结论

1. **核心收获只有一个:字段级跨源融合(#1)。** 我们现在是「多源比分→选一条最完整的」,本质是**整条择优**(和 MetaTube 一样);这个项目展示了更细的玩法——**把多源按字段特长融合成一条**,社区评分用社区源、官方标题/演员名用官方源、缺啥补啥。最终记录比任何单一源都完整准确。
2. 这与我们已有的 `detail_score` 排序**不冲突、可叠加**:用评分选出主源,再用其余源做字段级补全/覆盖。
3. 其余维度我们均已领先,无需借鉴。
