# 并行视频源查找 — 设计文档

日期：2026-06-23
状态：已与用户确认，待转实现计划

## 1. 背景与目标

「视频下载链接查找器」（`资源链接` 界面）当前一次只搜**一个**下载源，用户需手动下拉切换源逐个尝试。

目标：改成**并行查找**——同时跑多个源，每个源找到一条「有效正片」就停掉该源、把并行槽让给下一个待搜源，直到所有已启用源跑完或用户手动停止。最终给用户一份「每个有正片的源各一条」的结果供挑选下载。

### 已确认的需求决策

| 项 | 决策 |
|---|---|
| 结果模型 | **每源各收一条正片**，跑完所有源（不是"找到一个就全停"）。空出的并行槽接力给下一个待搜源。 |
| 并发数 | **最多 3 个**源并行（= 最多 3 个 webview）。做成设置 `linkFinderConcurrency`，默认 3。 |
| 单源预算 | 一个源最多搜 **~120 秒**还没出正片就判失败、关窗、换下一个。做成设置 `linkFinderSourceTimeoutSecs`，默认 120。页面明确 404 **立即**失败。 |
| 源范围/顺序 | **全部「已启用」的下载源**（设置里 enabled 的），顺序按现有列表；禁用的跳过。 |
| 停止条件 | 用户手动停止，**或**所有已启用源都搜完。 |
| 实现方案 | **A：前端调度 + 后端多窗口 finder**。 |

## 2. 复用的现有能力（不重写）

- **后端 `analyzeHls(url)`**：抓 m3u8 求时长/分辨率/isMaster/isVod。
- **前端"正片"判定** `realLink`（[VideoLinkFinder.vue](../../../src/components/VideoLinkFinder.vue) 现有逻辑）：HLS 链接 + `analyzeHls` 时长 **≥ 300 秒（5 分钟）**（滤掉 <120 秒广告/片段）+ 优先 master / 高分辨率。本设计将其抽成纯函数 `pickRealLink(links)` 按源各算一份。
- **后端拦截 JS** `INTERCEPT_JS`、CF 探测、404 检测、20 分钟硬上限兜底（[video_finder.rs](../../../src-tauri/src/resource_scrape/video_finder.rs)）。

## 3. 架构与数据流

```
前端调度器（队列 = 已启用源[列表序]，并发 = 3）
  └─ 每个槽：open_video_finder(code, site)
        后端开 video_finder_{site} 窗口，注入拦截 JS
        ├─ emit video-finder-link        {site, url}
        ├─ emit video-finder-page-state  {site, state:"not-found"}
        └─ emit video-finder-cf-state    {site, status, active}
  前端按 site 分流 → analyzeHls(url) → pickRealLink(该源 links)
        ├─ 出正片  → settle(found)、close(site)、放槽、pump 下一个源
        ├─ 404     → settle(notfound)、close、放槽、pump
        ├─ 120s 到 → settle(failed)、close、放槽、pump
        └─ CF      → 见 §6
  队列空且无活跃源 → 整体结束（scanning=false）
```

## 4. 后端改动（`src-tauri/src/resource_scrape/video_finder.rs`）

改动较小，核心是"单窗口 → 多窗口 + 事件打 site 标签"。

1. **窗口 label 改 per-site**：常量 `VIDEO_FINDER_LABEL = "video_finder_webview"` → 函数 `video_finder_label(site_id) = format!("video_finder_{site_id}")`。原"开窗前先关同 label"逻辑天然变成"只关该源自己的窗口"，支持并发多窗口。
   - 注意：`link_probe.rs` 复用了 `VIDEO_FINDER_LABEL`（仅 debug + env 触发）。改动时同步处理其引用（给它一个固定的 probe label 即可，二者互斥）。
2. **事件打 site 标签**：
   - `video-finder-link`：payload 从 `url: String` → `{ site: String, url: String }`。
   - `video-finder-page-state`：→ `{ site: String, state: String }`。
   - `video-finder-cf-state`：payload 增加 `site` 字段（前端按 site 分流 CF 状态）。
3. **command 调整**：
   - `open_video_finder_webview(code, site_id, show_on_cf)`：label/事件带 site；注入循环按 `video_finder_{site_id}` 是否存在判停（沿用现有）。
   - 新增/改 `close_video_finder_webview(site_id)`：关单个源窗口。
   - 新增 `close_all_video_finders()`：停止时关全部 `video_finder_*` 窗口。
4. 其余（`INTERCEPT_JS`、CF 探测、404 检测、20 分钟硬上限）**原样复用**，仅把发事件处带上 site。

## 5. 前端调度器（`src/composables/useParallelFinder.ts` + `VideoLinkFinder.vue`）

把调度逻辑抽成 composable，组件用它。

### 状态
- `queue: string[]`：已启用源（列表序）。
- `active: Map<siteId, SourceState>`：当前在搜的源（≤ concurrency）。
- `results: Map<siteId, SourceResult>`：已结算的源。
- `concurrency = 3`、`perSourceTimeout = 120s`（取自设置）。
- `SourceState`：`status`(pending/searching/cf/found/failed/notfound)、`links: VideoLink[]`、`realLink`、`timer`、`seen: Set<url>`、`startedAt`。

### 生命周期
- **启动**：`sites = 已启用源`；填满 3 个槽 → 每槽 `open(site) + 起 120s timer + 标记 searching`。
- **收链接**（`video-finder-link {site,url}`）：分流到该源 → 该源 `seen` 去重 → HLS 则 `analyzeHls` → 更新该源 `links` → `pickRealLink(links)`；出正片即 `settle(site,'found')`。
- **`settle(site, outcome)`**：清 timer、`close_video_finder_webview(site)`、写 `results`、从 `active` 删除、`pump()`。
- **`pump()`**：`active` 未满且 `queue` 非空 → 取下一个源开搜。
- **404**：`page-state{site,"not-found"}` 且该源尚无正片 → `settle(notfound)`。
- **120s 超时**：timer 触发 → `settle(failed)`。
- **停止**：取消所有 timer、`close_all_video_finders()`、清空 `active`/`queue`。
- **整体结束**：`active` 空且 `queue` 空 → `scanning=false`。
- **卸载（onUnmounted）/ 换番号**：等同停止，保证不漏窗口。

### 正片判定（复用）
抽 `pickRealLink(links): VideoLink | null` 为纯函数（即现有 `realLink` 逻辑），每源各算自己的 `links`。

## 6. CF 处理（并行下）

- 某源触发 CF 且**开启**「HTTP 失败回退 WebView」：该源窗口弹出居中供手动验证（后端现有 `listen_cf_visibility` 逻辑），状态 `CF验证中`，**该源 120 秒预算暂停**，改用现有 CF 手动超时（180s）为上限；通过 → 恢复抓取、状态回 `搜索中`，并**重置为新的 120 秒预算**（CF 过后才真正开始抓流，故给满额时间，而非接着旧计时）。
- **未开启**回退：CF 过不了 → 该源快速判 `failed`、立即让槽（不白占 120 秒）。
- v1 允许并行时同时弹多个 CF 窗口（同时撞 CF 少见），用户逐个验证；"一次只弹一个 CF" 作为后续优化（YAGNI）。

## 7. UI（`VideoLinkFinder.vue`）

- **顶栏**：番号输入 + 开始/停止 + 进度摘要（`已搜 X/Y 源 · 找到 Z 个正片`）。**移除单源下拉**。
- **主区：每个已启用源一行**（资源 N）：
  - 状态徽章：`等待中` / `搜索中`(转圈) / `CF验证中` / `✓ 正片` / `✗ 未找到` / `404`。
  - 找到正片时行内显示该正片：时长/分辨率徽章 + 链接 + `预览/下载/复制`。
  - **默认只显示该源那条正片**；可展开看该源其它捕获链接（滤掉广告/片段）。
- **底栏**：保存路径 + 批量下载（可多选多个源的正片一起加任务），沿用现有查重/下载逻辑。

## 8. 边界与错误处理

- 开窗失败 → 该源 `failed`、放槽。
- `analyzeHls` 失败 → 该链接不算正片，继续等其它/超时。
- 停止 / 卸载 / 换番号 → 关所有窗口 + 清 timer，**严格不漏窗口**。
- 并发**硬上限 3 个 webview**。
- 每源独立 `seen` 去重。
- 后端 20 分钟硬上限兜底（已有），防止某窗口异常长挂。

## 9. 设置（`settings`）

- `scrape.linkFinderConcurrency`：默认 3。
- `scrape.linkFinderSourceTimeoutSecs`：默认 120。
- 现有 `scrape.linkFinderSite`（单源选择）在并行模式下不再使用，保留以兼容旧配置即可（不删）。

## 10. 测试策略

webview 网络拦截难以自动化，务实分层：
- **`pickRealLink(links)`**（纯函数）→ Vitest 单测：给定链接集 → 选出正片（覆盖：无正片、多清晰度选 master/高分、广告/片段被排除、时长边界 300s）。
- **调度器纯逻辑**（取下一个源、并发上限、`settle` 状态机、整体结束判定）→ Vitest 单测（可把后端调用注入为 mock）。
- **后端 label/事件改动**：机械改动，靠 `cargo check` + 现有 `build_site_url` 测试守住。
- **webview 集成**：手动真机验证——搜一个番号，观察 ≤3 个窗口并行、每源出正片、404/超时正确让槽、停止关闭全部窗口、CF 暂停预算。

## 11. 不做（YAGNI）

- "一次只弹一个 CF 窗口"的排队（v1 允许堆叠）。
- 跨源去重/合并同一条正片（不同源同一 m3u8 极少见）。
- 自动下载"最佳源"（仍由用户挑选下载）。
- 源的自定义排序/每次勾选子集（按已启用 + 列表序）。
